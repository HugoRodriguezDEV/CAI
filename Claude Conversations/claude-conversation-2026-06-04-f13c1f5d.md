# Flujo de trabajo con Git

Aplica estas reglas cada vez que el usuario necesite realizar cualquier operación de Git en este proyecto.

---

## Convención de Commits

**Formato obligatorio:**

```
<prefijo>: <descripción en español, primera persona pasado informal> #<número-issue>
```

### Prefijos disponibles

| Prefijo  | Cuándo usarlo |
|----------|---------------|
| `add:`   | Nuevo código, archivo o funcionalidad |
| `fix:`   | Corrección de bug o comportamiento incorrecto |
| `refac:` | Reestructura sin cambio de comportamiento observable |
| `doc:`   | Solo documentación (README, CLAUDE.md, docs/) |

### Reglas del mensaje

- Descripción en **español informal**, primera persona, tiempo pasado.
  - Correcto: `agrege`, `cree`, `corregi`, `actualice`, `elimine`, `movi`
  - Incorrecto: `agregué` (con tilde), `add service`, `se agrego`
- **Minúsculas** siempre, excepto siglas o nombres propios de código (`SAAF`, `RTK`, `RTKQuery`).
- Sin punto final.
- La referencia `#<N>` es **obligatoria** cuando el commit pertenece a una issue; omítela solo si es un cambio de mantenimiento menor sin issue asociada.
- Mantén la descripción concisa pero suficientemente descriptiva para entender el cambio sin leer el diff.

### Ejemplos válidos

```bash
git commit -m "add: agrege servicio de autenticacion SAAF #5"
git commit -m "fix: corregi tipos en la respuesta de login #3"
git commit -m "refac: movi los tipos de saaf a su propio directorio #1"
git commit -m "doc: actualice README con seccion de git workflow"
git commit -m "add: cree hook useFactors con soporte de busqueda en memoria #7"
```

### Ejemplos inválidos

```bash
# ❌ En inglés
git commit -m "add: added authentication service #5"

# ❌ Con tilde
git commit -m "add: agregué el servicio de autenticación #5"

# ❌ Prefijo incorrecto
git commit -m "feat: agrege servicio de autenticacion #5"

# ❌ Sin prefijo
git commit -m "agrege servicio de autenticacion #5"
```

## Convención de Ramas

### Nomenclatura

```
<tipo>/issue-<N>-<descripcion-en-kebab-case>
```

| Tipo      | Cuándo usarlo |
|-----------|---------------|
| `feature` | Nueva funcionalidad ligada a una issue |
| `fix`     | Corrección de bug ligada a una issue |
| `hotfix`  | Corrección urgente en producción (sin issue previa) |
| `refac`   | Refactorización planificada |
| `doc`     | Solo documentación |
| `chore`   | Mantenimiento (deps, config, CI) |

### Ejemplos

```bash
feature/issue-1-redux-toolkit-rtk-query
fix/issue-12-error-en-login-saaf
hotfix/token-expiration-crash
doc/issue-9-actualizar-readme-servicios
```

## Comandos Esenciales

### Crear y publicar una nueva rama

```bash
# Desde main siempre actualizado
git checkout main
git pull origin main

# Crear y cambiar a la nueva rama
git checkout -b feature/issue-<N>-<descripcion>

# Publicar la rama en el remoto por primera vez
git push -u origin feature/issue-<N>-<descripcion>
```

### Guardar cambios (commit)

```bash
# Ver estado actual
git status

# Agregar archivos específicos (preferir sobre git add .)
git add src/services/auth.ts src/types/_auth.type.ts

# Crear commit con la convención del proyecto
git commit -m "add: agrege servicio y tipos de autenticacion SAAF #5"
```

### Actualizar rama con cambios remotos

```bash
# Traer cambios sin fusionar (ver qué hay)
git fetch origin

# Actualizar tu rama actual
git pull origin <nombre-rama>

# Actualizar main y hacer rebase de tu rama sobre él
git checkout main && git pull origin main
git checkout feature/issue-<N>-<descripcion>
git rebase main
```

### Push de cambios

```bash
# Push normal (rama ya tiene tracking)
git push

# Push forzado después de un rebase (solo en ramas propias, nunca en main)
git push --force-with-lease
```

## Merges — Cerrar una Issue con PR

El flujo recomendado para integrar trabajo terminado es siempre a través de un **Pull Request** en GitHub.

### Flujo completo para cerrar una issue

```bash
# 1. Asegúrate de que tu rama está al día con main
git fetch origin
git rebase origin/main

# 2. Empuja los últimos cambios
git push

# 3. Crea el PR desde GitHub CLI
gh pr create \
  --title "feat: <descripcion corta>" \
  --body "Closes #<N>" \
  --base main \
  --head feature/issue-<N>-<descripcion>
```

### Merge local (solo si no se usa PR)

```bash
# Asegúrate de estar en main y tenerlo actualizado
git checkout main
git pull origin main

# Merge con --no-ff para preservar el historial de la rama
git merge --no-ff feature/issue-<N>-<descripcion> -m "add: merge issue #<N> — <descripcion corta>"

# Publica en el remoto
git push origin main

# Elimina la rama local ya integrada
git branch -d feature/issue-<N>-<descripcion>

# Elimina la rama remota
git push origin --delete feature/issue-<N>-<descripcion>
```

## Crear una Issue en GitHub

```bash
# Con GitHub CLI
gh issue create \
  --title "feat: <Titulo descriptivo>" \
  --body "<Descripcion detallada siguiendo el formato de /issue-role>" \
  --label "enhancement"
```

Para redactar el contenido del issue de forma estructurada, usa la skill `/issue-role` antes de ejecutar este comando.

## Actualizar una Rama con Cambios de Main (Rebase vs Merge)

Preferir **rebase** sobre **merge** para mantener historial lineal:

```bash
# Rebase recomendado
git fetch origin
git rebase origin/main

# Solo si hay conflictos en el rebase
git rebase --continue   # después de resolver cada conflicto
git rebase --abort      # para cancelar y volver al estado previo
```

Usar **merge** solo cuando la rama es compartida con otros desarrolladores y el rebase reescribiría commits ya publicados.

## Comandos de Inspección Útiles

```bash
# Ver historial limpio con grafo
git log --oneline --graph --all

# Ver diferencias antes de hacer commit
git diff

# Ver diferencias de archivos en staging
git diff --staged

# Ver quién modificó cada línea
git blame src/index.ts

# Buscar en el historial
git log --oneline --grep="SAAF"

# Ver ramas locales y remotas
git branch -a

# Ver estado de la rama vs remoto
git status -sb
```

## Publicación a Producción (Releases)

Para publicar una nueva versión de la librería, se debe de ejecutar este flujo y con ello, asegurar que el repositorio, los tags y el gestor de paquetes mantengan la misma versión documentada.

### Flujo Completo de Publicación

```bash
# 1. Asegúrate de estar en la rama principal y que esté actualizada
git checkout main
git pull origin main

# 2. Genera un nuevo empaquetado para revisar que todo logre compilar sin errores
npm run build

# 3. Actualiza la versión en el package.json rigiéndose del versionado Semántico
# El comando "npm version" modificará package.json, realizará el commit y creará el tag automáticamente.
#   - patch: para correcciones y parches sin modificar API (ej. 1.0.0 a 1.0.1)
#   - minor: nuevas funciones sin romper retrocompatibilidad (ej. 1.0.0 a 1.1.0)
#   - major: cambios bruscos o importantes que exigen adaptación y rompen el código viejo (ej. 1.0.0 a 2.0.0)
npm version minor -m "chore: bump version to %s"

# 4. Sube los últimos cambios realizados a main (incluyendo el commit automatizado del paso anterior)
git push origin main

# 5. Sube los nuevos tags registrados en tu local producto de npm version
git push origin --tags

# 6. Sube y publica la versión de la librería al registry restringido de GitHub Packages
npm publish
```

### Reglas sobre Tags (Versionamiento manual alternativo)

En caso de que el flujo general haya fallado o requieras control manual para un tag pre-establecido (sin npm):
```bash
# Crear tag de versión manual
git tag -a v1.2.0 -m "release: version 1.2.0"

# Publicar unicamente tu tag
git push origin v1.2.0
```

## Caso de Uso - Publicando la Librería

Cuando el usuario requiera asistencia con lanzar su código o el proceso a producción, a esta secuencia te deberás atener para sugerir y realizar los comandos:

1. **Última Verificación:** Finalizaron todas las tareas y estás situado en la rama `main` actualizada con `git pull origin main`. Te cercionas del estado base ejecutando `npm run build`.
2. **Bumping & Tags:** Solicitas u ocurres a usar el incrementeo oficial con npm en la raíz ejecutando `npm version minor -m "chore: bump version to %s"` y procedes mandándolo a la red remota tanto al branch `git push origin main`, como también con su respectivo tag `git push origin --tags`.
3. **Publicación:** Posteriormente, invocas en la raíz del proyecto `npm publish` y observas los logs de publicación notando que el release del código ha concluido hacia el registry remoto de GitHub con éxito.
